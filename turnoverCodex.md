# turnoverCodex

## 1. Project Summary

Paradisio is a static, mobile-friendly business directory and community board for Puerto Viejo de Talamanca, Costa Rica and nearby areas. It aggregates local businesses within roughly 5 km of Puerto Viejo, enriches them with contact/social/map data, and publishes a GitHub Pages app under `docs/paradisio_app/`.

Primary users:
- Visitors and residents looking for hotels, rentals, restaurants, shops, tours, services, and local classifieds.
- Project maintainers/agents enriching a local business dataset without paid APIs or Google API keys.
- Business owners who may eventually claim/correct listings.

Current state:
- The current unified master file is `pv_master_unified.csv`, with 750 rows and 34 columns.
- The generated app has 750 business detail pages, 15 classifieds listings, 16 classifieds HTML pages including index, 751 QR PNGs, and a static directory/map UI.
- The app is generated by `paradisio_app/build.py` into `docs/paradisio_app/`.

Older docs still mention 450 or 615 records in places; the actual current generated app and `pv_master_unified.csv` are at 750 records.

## 3. Architecture

This is a static site generator plus generated static app. There is no backend server, database, or framework at runtime.

Main source app:
- `paradisio_app/build.py`: generator.
- `paradisio_app/static/app.js`: directory frontend behavior.
- `paradisio_app/static/classifieds.js`: classifieds search behavior.
- `paradisio_app/static/tokens.css`: design tokens.
- `paradisio_app/static/styles.css`: app styles.
- `paradisio_app/data/classifieds.json`: source classifieds data.
- `paradisio_app/maps_enrich.py`: Maps enrichment crawler.
- `paradisio_app/generate_qr.py`: QR generation support.
- `paradisio_app/capture_mobile.py`: screenshot/QA support.

Generated output:
- `docs/paradisio_app/index.html`: main directory.
- `docs/paradisio_app/businesses/*.html`: one business page per record.
- `docs/paradisio_app/classifieds/*.html`: classifieds board and detail pages.
- `docs/paradisio_app/static/*`: copied JS/CSS.
- `docs/paradisio_app/data/*.json`: generated app data and enrichment.
- `docs/paradisio_app/qr/*.png`: QR assets.

Generator flow in `build.py`:
1. Reads `pv_master_unified.csv`.
2. Reads `docs/paradisio_app/data/maps_enrich.json` if present.
3. Reads `paradisio_app/data/classifieds.json`.
4. Builds normalized business objects:
- cleaned display name
- slug
- deterministic ID
- contact channels
- primary contact
- secondary links
- scores
- badges
- intents
- Maps enrichment fields
5. Writes `businesses.json` and `metrics.json`.
6. Generates 750 business detail HTML pages.
7. Generates directory index.
8. Generates classifieds pages if classifieds exist.
9. Copies static JS/CSS into generated docs directory.

Frontend behavior in `app.js`:
- Uses global embedded constants `BUSINESSES`, `CATEGORIES`, and `AREAS`.
- Provides text search across name, category, area, description, badges.
- Filters by category, area, and contact channel.
- Sorts by name, contactability, or completeness.
- Renders paginated list cards, 50 per page.
- Supports “load more”.
- Supports category shortcut tiles.
- Supports list/map toggle.
- Map uses Leaflet plus markercluster from CDN.
- Markers are filtered based on current filters.

Design system:
- `tokens.css` defines colors, typography scale, spacing, radii, shadows, max width, nav height.
- Visual palette is sand/surface, jungle green, reef teal, coral, sun yellow.
- `styles.css` implements mobile-first layout, nav, masthead, stats, filters, cards, map, business pages, sticky mobile CTA, QR blocks, scores, classifieds, and responsive rules.

Bridge/agent system:
- `AGENTS.md` defines project rules and delegation protocols.
- `codex_bridge.py` is the root wrapper for Codex CLI.
- `CODEX_ENDPOINT/` is a file-based IPC hub.
- v1 one-shot: `python codex_bridge.py "task" --json`.
- v2 session flow uses `CODEX_ENDPOINT/session_orchestrator.py`.
- Sessions live in `CODEX_ENDPOINT/sessions/`.
- Artifacts live under `CODEX_ENDPOINT/tasks/<session_id>/artifacts/`.
- Responses/audit logs live under `CODEX_ENDPOINT/responses/bridge_*.json`.
- Session files are schema-validated, append-only, atomically written, lock-protected, and state-machine controlled.

## 5. Risks & Issues

Data consistency risks:
- `AGENTS.md` still says canonical file is `pv_within_5km_enriched_b.csv` with 450 records, but the app and README use `pv_master_unified.csv` with 750 records. New agents should treat `pv_master_unified.csv` as current app master unless explicitly told otherwise.
- Several planning docs are stale and refer to 615 or 450 records.
- Generated `businesses.json` uses cleaned display name for `name`, but `primary_contact.url` text can still include the raw CSV business name with location suffixes because `get_primary_contact(row)` runs before the cleaned name is substituted.
- Some records have coordinates missing but CIDs present.
- 131 records are Uncategorized and area Unknown.
- Instagram fields are mixed: `instagram_confidence`, `instagram_enrich_confidence`, and `ig_verified` can disagree or be blank.
- `has_whatsapp()` treats normalized phone as WhatsApp when `whatsapp` is blank, so current “with WhatsApp” count may mean “has a phone usable in wa.me format,” not verified WhatsApp.
- Maps enrichment contains Spanish UI noise in amenities/prices and repeated price strings; `biz_amenities()` filters some noise but not all possible noise.
- `maps_enrich.json` success means page extraction succeeded, not necessarily that every extracted field is correct.

Frontend/app risks:
- Business detail pages always render “Google Maps verified” and “Instagram verified” trust badges in `.biz-trust`, regardless of whether that specific business has Maps/Instagram verification. This should be made conditional.
- `nav_html()` defines Spanish labels but currently only outputs English labels.
- Category shortcut mapping is lossy: “stay” maps only to `hotel`, not hostel/vacation_rental; “nightlife” maps to restaurant; “transport” maps to services.
- Leaflet and markercluster are CDN dependencies; offline or CDN failures break map behavior.
- HTML is assembled with f-strings. Most frontend list rendering escapes JS content, but generated detail pages interpolate fields directly into HTML attributes/text. Data should be escaped centrally before production hardening.
- Generated pages include `data-plausible-*` attributes while analytics script is GoatCounter; event tracking may not be wired as intended.
- Placeholder email `paradisio@example.com` appears in claim/post flows and should be replaced before public use.

Operational risks:
- Google Maps scraping is fragile and subject to anti-automation/CAPTCHA. Existing docs emphasize pacing and direct CID loads.
- Browser automation requires `danger-full-access` on this Windows machine.
- No paid APIs or commercial datasets are allowed.
- Do not modify master datasets unless explicitly instructed.
- Do not recursively delegate from Codex to OpenCode.
- Generated files under `docs/paradisio_app/` can be overwritten by `build.py`; edit source files in `paradisio_app/` unless intentionally hotfixing generated output.
- `CODEX_ENDPOINT/responses/` and task artifacts contain many historical logs, some very large. Avoid treating them as source of truth unless investigating delegation history.
- `screenshots/` contains many large PNG map captures; useful evidence, but bulky.

2. Treat `pv_master_unified.csv` as the current app master and inspect a few rows.
3. Read `README.md` for the current public-facing description, but remember older counts in other docs may be stale.
4. Read `paradisio_app/build.py` end to end. It is the core of the app.
5. Read `paradisio_app/static/app.js`, `classifieds.js`, `tokens.css`, and `styles.css`.
6. Inspect `docs/paradisio_app/data/businesses.json`, `metrics.json`, and `maps_enrich.json` to understand generated runtime data.
7. Inspect `paradisio_app/data/classifieds.json` and `docs/paradisio_app/classifieds/`.
8. Inspect `docs/paradisio_app/qr/` only by count/sample; there are hundreds of images.
9. For enrichment tasks, read `paradisio_app/maps_enrich.py`, `stealth_search.py`, `website_crawl.py`, `crossref_osm.py`, `geofilter.py`, and `pvscraper/`.
10. For delegation tasks, read `codex_bridge.py`, `CODEX_ENDPOINT/README.md`, `session_orchestrator.py`, `session_lib.py`, and `session_schema.py`.

`python paradisio_app/maps_enrich.py`

`python CODEX_ENDPOINT\session_orchestrator.py create --title "My Task" --description "Do X"`

Working rules for new agents:
- Never read or rely on generated prose as canonical when code/data disagree.

- Before changing data, confirm whether the user means `pv_master_unified.csv` or older `pv_within_5km_enriched_b.csv`.
- Preserve CSV schema unless asked to migrate it.
- Add new enrichment outputs to assigned files first; merge into master only with explicit instruction.
- Validate app changes by rebuilding and checking generated counts/pages.
- Be cautious with Google/Maps automation; respect pacing and use direct CID loads where possible.
- Replace placeholder public contact values before any production claim/classifieds workflow is promoted.
